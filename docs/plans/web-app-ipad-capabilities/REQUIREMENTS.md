# Web / iPhone / iPad 能力规划需求文档

## 1. 背景与定位

当前 `korean_learn` 已经不是一个单纯资料展示站，而是一个面向中文母语韩语初学者的训练工具。现有 Web 版本已经具备：

- 音标发音训练：四十音、收音、双收音、示范词、发音技巧、播放队列。
- 学习路径：给用户学习思路，而不是强制生成计划。
- 场景句子：按场景查看句子，支持音频播放、慢速和循环。
- 词汇：通过例句理解词汇，不做孤立词表。
- 教材：基于对象存储加载教材页图和音频，PDF 不再作为前端主阅读资源。
- 后台管理：管理场景、句子、教材内容。

下一阶段规划应遵循一个原则：

```text
一套功能模型，一套用户数据，多端不同布局。
```

也就是说，Web、iPhone、iPad 不应该是三个互相割裂的产品。用户在任意端产生的学习记录、收藏、笔记、批注、进度，都应该进入同一套学习数据。

## 2. 外部产品调研结论

本次参考了语言学习类和 iPad 学习工具类产品。调研重点不是照搬功能，而是提炼高频产品模式。

### 2.1 Duolingo：短练习和习惯机制

Duolingo 的核心启发是：

- 用短练习降低启动成本。
- 用 streak、XP、排行榜、提醒提高复访。
- 课程路径清晰，用户不用每次决定“今天学什么”。

对本项目的启发：

- 可以借鉴“每日短练习”，但不建议过度游戏化。
- 重点不是排行榜，而是让用户每天完成一个小闭环：听、读、跟读、复习。
- 保持安静学习工具气质，不做强刺激的游戏界面。

### 2.2 LingoDeer：适合日韩学习的结构化路径

LingoDeer 对韩语、日语等语言的价值在于：

- 清晰的学习路径。
- 语法解释和 tips。
- 短课时、复习、测验。
- 原生语音和跟读模仿。

对本项目的启发：

- 韩语学习不能只堆词汇，需要解释音标、收音、句型和语法。
- 四十音详情页应该像“训练说明卡”，而不是只展示词。
- 教材学习应逐步结构化为“页码、课、音频、词汇、句型、练习”。

### 2.3 Babbel / Busuu：真实场景、发音反馈、复习系统

Babbel 和 Busuu 的共同方向：

- 以真实对话和生活场景组织内容。
- 使用语音识别或录音反馈帮助口语练习。
- 有复习系统和进度追踪。
- Busuu 还强调社区/真人反馈。

对本项目的启发：

- 场景句子模块应从“句子列表”升级为“场景练习”。
- 录音跟读是 App 端最值得优先规划的能力之一。
- 第一阶段不做复杂 AI 评分，先做录音、回放、对比和自评。

### 2.4 Memrise / Teuida：真实语音和开口练习

Memrise 强调真实母语者视频和沉浸输入，Teuida 强调第一人称场景开口练习。

对本项目的启发：

- 音频质量非常重要，TTS 只能做兜底，核心内容应优先使用真人或高质量素材。
- App 端应强化“听一句、跟一句、回放对比”的训练闭环。
- 不要只让用户看教材，要让用户开口。

### 2.5 Goodnotes / Notability：iPad 教材工作台

iPad 学习工具的核心能力：

- PDF/图片阅读。
- Apple Pencil 手写批注。
- 划线、高亮、备注。
- 页面级笔记。
- 搜索和整理。

对本项目的启发：

- iPad 端适合做教材工作台。
- 初期不必依赖复杂 PDF 语义解析，页码 + 坐标就能实现批注、划线和备注。
- 教材结构化数据后续再支持逐句播放、句子高亮和生词联动。

## 3. 产品总原则

### 3.1 功能不分端，布局适配端

同一功能在三端都应该存在：

- 音标训练。
- 词汇学习。
- 场景句子。
- 教材学习。
- 播放、循环、慢速。
- 收藏。
- 笔记。
- 进度。
- 复习。

区别只在交互布局：

- Web：适合快速访问、管理内容、验证功能。
- iPhone：适合碎片训练、复习提醒、录音跟读。
- iPad：适合教材阅读、批注、分栏学习、深度整理。

### 3.2 数据必须统一

未来所有端共享同一套用户学习数据：

- 学习进度。
- 每日练习记录。
- 收藏词汇/句子/教材页。
- 错题或薄弱项。
- 跟读录音记录。
- 教材笔记。
- 页面批注。
- 复习队列。

### 3.3 先做训练闭环，再做内容规模

当前项目不应追求“资料库很大”，而应优先做好：

```text
选择一个学习单位
听原音
看解释
跟读
记录
复习
```

## 4. 三端能力规划

### 4.1 Web 端

Web 的定位：

```text
学习入口 + 内容管理 + 功能验证 + 轻量学习
```

Web 应保留并增强：

- 学习路径。
- 音标训练。
- 词汇和例句。
- 场景句子。
- 教材页图阅读。
- 教材音频播放。
- 后台内容管理。

Web 不优先承担：

- 重度手写批注。
- 长时间录音训练。
- 离线学习。
- 推送提醒。

Web 重点新增能力：

- 登录后同步学习进度。
- 收藏词汇、句子、教材页。
- 查看历史练习记录。
- 管理教材结构化数据。
- 后台维护音频、示范词、课文时间轴。

### 4.2 iPhone App

iPhone 的定位：

```text
每日训练 + 碎片复习 + 跟读反馈
```

iPhone 应具备完整学习功能，但以单栏任务流呈现：

- 首页显示“继续学习”和“今日训练”。
- 音标训练支持听音、示范词、播放次数。
- 词汇复习使用卡片或小测。
- 场景句子支持逐句播放、慢速、循环。
- 教材学习可以阅读页图，但不作为最强场景。
- 录音跟读支持录音、回放、与原音对比。
- 离线音频包支持基础音标和常用词句。
- 提醒用户进行每日练习。

iPhone 第一优先级：

- 每日 5 分钟训练。
- 复习队列。
- 录音跟读。
- 收藏和进度同步。

### 4.3 iPad App

iPad 的定位：

```text
教材工作台 + 深度学习 + 批注整理
```

iPad 使用同一套功能模型，但用大屏布局：

- 左侧教材页图。
- 右侧音频、词汇、笔记、句子。
- 支持 Apple Pencil 批注。
- 支持划线、高亮、备注。
- 支持按页保存笔记。
- 支持教材页和音频段落联动。
- 支持章节目录和页码跳转。
- 支持将收藏词汇进入复习队列。

iPad 第一阶段不强依赖 PDF 语义解析。批注可以基于：

```text
book_id + page_number + normalized_page_coordinates
```

后续如果要做“点击句子播放音频”或“课文逐句高亮”，需要增加教材结构化数据：

```text
lesson
page
sentence
audio_url
audio_start
audio_end
bounding_box
vocabulary
grammar_points
```

## 5. 统一功能模型

### 5.1 学习单元

统一抽象为 `learning_item`，可对应：

- 音标。
- 示例词。
- 词汇。
- 句子。
- 场景。
- 教材页。
- 教材句子。
- 音频片段。

每个学习单元都可以支持：

- 播放。
- 收藏。
- 记录学习次数。
- 加入复习。
- 关联笔记。

### 5.2 进度模型

需要记录：

- 用户当前学习到哪里。
- 每个模块完成度。
- 最近学习时间。
- 听音次数。
- 跟读次数。
- 复习状态。

最低可行字段：

```text
user_id
item_type
item_id
status
practice_count
last_practiced_at
next_review_at
```

### 5.3 收藏模型

收藏对象包括：

- 音标。
- 词汇。
- 句子。
- 教材页。
- 教材音频段落。

收藏不是独立功能，而是复习入口。

### 5.4 笔记模型

笔记分三类：

- 文本笔记：任何端都可以创建。
- 页面备注：绑定教材页。
- 手写/划线批注：主要在 iPad 创建，但 Web 和 iPhone 可以查看。

页面批注需要使用归一化坐标：

```json
{
  "book_id": "yonsei1",
  "page": 12,
  "type": "highlight",
  "rects": [
    { "x": 0.18, "y": 0.42, "w": 0.36, "h": 0.03 }
  ]
}
```

手写批注：

```json
{
  "book_id": "yonsei1",
  "page": 12,
  "type": "ink",
  "strokes": [
    {
      "color": "#222222",
      "width": 2,
      "points": [
        { "x": 0.12, "y": 0.31 },
        { "x": 0.13, "y": 0.32 }
      ]
    }
  ]
}
```

### 5.5 跟读模型

第一阶段不做复杂 AI 评分，先做：

- 原音播放。
- 用户录音。
- 回放对比。
- 用户自评。
- 保存最近几条录音。

后续再升级：

- 语音识别。
- 发音相似度。
- 音素级反馈。
- 教练建议。

## 6. 推荐阶段规划

### Phase 1：Web 数据层补齐

目标：为 App/iPad 打基础。

需要做：

- 增加账号登录。
- 增加用户学习进度。
- 增加收藏。
- 增加笔记。
- 增加复习记录。
- 整理教材结构化字段。

验收：

- Web 登录后可以保存收藏和进度。
- 换浏览器后数据仍能恢复。
- 音标、词汇、句子、教材页都能进入用户资产。

### Phase 2：iPhone App 原型

目标：验证移动端训练闭环。

需要做：

- 一个 Universal App 项目，先跑 iPhone 布局。
- 接入 Web 后端 API。
- 实现登录、音标训练、词汇复习、场景句子。
- 实现录音和回放。
- 实现本地缓存基础音频。

验收：

- 用户能在手机上完成一次 5 分钟训练。
- 训练结果同步回 Web。
- 收藏和复习记录跨端一致。

### Phase 3：iPad 教材工作台

目标：验证大屏深度学习。

需要做：

- 同一个 App 增加 iPad 双栏布局。
- 教材页图阅读。
- 音频面板。
- 页面笔记。
- Apple Pencil 批注。
- 划线和备注。

验收：

- 用户可以在 iPad 打开教材，播放对应页音频。
- 用户可以在教材页上划线、写备注、手写批注。
- 批注可同步，并能在 Web 或 iPhone 查看。

### Phase 4：教材结构化联动

目标：把教材从“阅读材料”变成“训练材料”。

需要做：

- 教材课文句子结构化。
- 句子和音频时间轴绑定。
- 句子和教材页面坐标绑定。
- 点击句子播放音频。
- 逐句跟读。
- 生词自动进入复习队列。

验收：

- iPad 点击教材句子可以播放对应音频片段。
- App 能复习教材中收藏的词句。
- Web 后台能维护课文、音频、页码和坐标。

## 7. 技术建议

### 7.1 App 技术路线

建议优先考虑：

```text
React Native / Expo
```

原因：

- 一个项目同时支持 iPhone 和 iPad。
- 未来如果需要 Android，也有扩展空间。
- 可以复用一部分 Web 的数据模型和业务理解。
- 适合先做原型。

如果明确只做 Apple 生态，并且非常重视 Apple Pencil、iPad 原生体验，也可以考虑：

```text
SwiftUI
```

建议判断：

- 第一版验证产品：React Native / Expo。
- 后续重度 iPad 笔记体验：评估 SwiftUI 或原生模块。

### 7.2 后端演进

当前后端是 Python 原生 `http.server` + MySQL，适合教学和小型验证。三端同步后，建议逐步升级：

- 保留 MySQL。
- 增加用户认证。
- 增加 REST API。
- 增加对象存储资源管理。
- 后续考虑 Flask / FastAPI。

不要一开始重构整个后端。优先补数据模型和 API，再决定框架升级。

### 7.3 资源策略

教材、音频、页图继续放对象存储。

App 端缓存策略：

- 基础音标音频可内置或首启缓存。
- 教材音频按课下载。
- 教材页图按章节缓存。
- 用户录音可先本地保存，登录后再同步。

## 8. 近期不建议做的事

- 不建议立即做完整 AI 发音评分。
- 不建议一开始做社区功能。
- 不建议把 iPhone 和 iPad 拆成两个 App。
- 不建议在 App 第一版里塞完整教材结构化。
- 不建议用排行榜替代真实学习反馈。
- 不建议让用户一打开就面对大量资料列表。

## 9. 第一版 App / iPad MVP

### 必须有

- 登录。
- 音标训练。
- 示例词播放。
- 词汇复习。
- 场景句子播放。
- 收藏。
- 学习进度。
- 录音跟读。
- iPad 教材页图阅读。
- iPad 页面笔记。

### 可以延后

- Apple Pencil 高级笔刷。
- AI 发音评分。
- 逐句教材高亮。
- 社区反馈。
- 排行榜。
- 多教材市场。
- 复杂学习计划。

### 成功标准

第一版成功不是“功能最多”，而是用户能形成一个清楚的学习闭环：

```text
今天打开
继续上次内容
听一个音/词/句
跟读
收藏不熟的内容
下次复习
```

## 10. 关键产品判断

1. 你的产品不应该变成 Duolingo。
   - 可以借鉴短练习和习惯机制。
   - 不建议借鉴强游戏化和排行榜压力。

2. 你的产品不应该变成纯 Goodnotes。
   - 可以借鉴批注和教材工作台。
   - 但核心仍然是韩语训练，而不是通用笔记。

3. 你的产品不应该变成资料库。
   - 内容要服务训练。
   - 每个内容都应该能被听、读、跟读、收藏、复习。

4. 三端功能保持一套是正确方向。
   - 分端的是布局，不是功能资产。
   - 用户学习数据必须统一。
